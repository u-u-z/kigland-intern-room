# 发现 → Issue 转化流程

> 当我在漫游中发现值得深入的东西，立即转化为可追踪的 Issue。

---

## 触发条件（当以下情况发生时）

### 🔥 立即创建 Issue（高优先级）

**1. HN 热门技术突破**
- 条件：HN 上超过 500 赞 + 技术相关
- Issue 标题：`[热点追踪] X 技术/工具发布`
- 内容模板：
  ```markdown
  ## 发现
  - HN 链接：
  - 原文链接：
  - 当前热度：X 赞，Y 评论
  
  ## 为什么值得关注
  - （一句话总结突破性）
  
  ## 建议行动
  - [ ] 快速了解核心概念
  - [ ] 判断是否值得深入研究
  - [ ] 如有必要，启动专项研究
  ```

**2. 高质量新信源**
- 条件：发现持续产出高质量内容的网站
- Issue 标题：`[信源发现] X 网站/博客`
- 内容模板：
  ```markdown
  ## 网站信息
  - 域名：
  - 类型：个人博客/公司工程博客/开源项目/研究机构
  - 主要内容方向：
  
  ## 样本文章（证明质量）
  - 标题1：
  - 标题2：
  
  ## 建议
  - 是否加入每日检查清单：是/否
  - 优先级：P0/P1/P2
  ```

**3. 系统性问题或机会**
- 条件：发现可以优化的流程、缺失的工具、有趣的模式
- Issue 标题：`[系统改进] X 问题/机会`
- 内容模板：
  ```markdown
  ## 问题/机会描述
  - 
  
  ## 影响
  - 
  
  ## 建议方案
  - 
  
  ## 需要决策
  - @Remi 请确认是否值得投入
  ```

---

### 📋 记录并定期批量创建（中优先级）

**每日发现汇总**
- 每天晚上回顾当天的漫游记录
- 把「值得关注但不够紧急」的发现汇总
- 批量创建 Issues 或追加到已有 Issue

**周期性深度发现的课题**
- 当某个领域积累足够多素材
- 创建 `[研究课题] X 领域的体系化梳理`

---

## Issue 标签体系

**类型标签**:
- `research` - 研究课题
- `source` - 信源发现
- `tool` - 工具/技能
- `improvement` - 系统改进
- `hot` - 热门追踪（紧急）

**状态标签**:
- `status:ready` - 准备好可以开始
- `status:triage` - 需要评估/决策
- `status:wip` - 进行中
- `status:waiting` - 等待外部输入

**领域标签**:
- `area:ai` - 人工智能
- `area:systems` - 系统编程
- `area:web` - Web 技术
- `area:tools` - 开发工具
- `area:security` - 安全

---

## 主动推送 vs 创建 Issue 的决策树

```
发现有趣内容
    ↓
是紧急/热门内容？（HN 500+ 赞，重大发布）
    ├── 是 → 立即发消息给 Remi + 创建 Issue
    ↓
是需要长期追踪的课题？
    ├── 是 → 创建 Issue，添加到研究议程
    ↓
是高质量信源？
    ├── 是 → 创建 Issue，更新 sources.md
    ↓
是系统改进机会？
    ├── 是 → 创建 Issue，等待 Remi 决策
    ↓
以上都不是，但有趣？
    └── 记录到每日笔记，晚上汇总时决定
```

---

## 创建 Issue 后的动作

**立即执行**:
1. 在 Issue 中 @Remi
2. 添加合适的标签
3. 更新 `agenda.md`（如果是研究课题）
4. 更新 `sources.md`（如果是信源发现）

**后续跟进**:
- 如果 Remi 回复，根据反馈调整优先级
- 如果标记为 `status:ready`，开始执行
- 定期更新进展，直到完成
